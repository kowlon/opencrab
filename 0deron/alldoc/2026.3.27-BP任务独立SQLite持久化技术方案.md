# BP 任务独立 SQLite 持久化技术方案

> 文档日期：2026-03-27
> 状态：待实施

***

## 1. 背景与问题

当前 BP（Best Practice）实例状态通过 `session.metadata["bp_state"]` → `sessions.json` 文件持久化。存在以下问题：

1. **Session 过期即丢失**：BP 历史记录随 session 生命周期消亡，无法跨 session 查询已完成/已取消的任务
2. **持久化有延迟**：sessions.json 写入采用 debounced 5s 机制，前端收到 SSE 事件后立即查询可能拿不到最新数据
3. **数据不独立**：BP 状态嵌套在 session.metadata 中，无法独立查询、筛选、统计
4. **前端显示层能力受限**：缺少跨 session 列表查询、按状态筛选、单实例详情等端点

## 2. 目标

将 BP 实例数据存入独立 SQLite 表，提供 **实时更新** 和 **丰富查询** 能力：

- 显示层可随时查询任务信息用于展示
- 支持对具体 subtask output 的指定字段进行编辑并持久化
- 支持触发下一步（bp\_next / bp\_answer）时数据始终一致
- 支持服务重启后完整恢复切换/挂起状态

## 3. 设计原则

| 原则             | 说明                                                                         |
| -------------- | -------------------------------------------------------------------------- |
| **写入同步 await** | 每次状态变更后立即 `await` 写入 SQLite，不用 fire-and-forget。前端在收到 SSE 事件后查询，必须能拿到最新数据   |
| **读取双源**       | 内存优先（快），内存无数据时从 SQLite 恢复（server restart 场景）                               |
| **查询丰富**       | 新增跨 session、按状态、按 bp\_id 的查询端点，支持前端任务列表/历史/详情展示                            |
| **字段级更新**      | `bp_edit_output` 只更新被编辑的 subtask output，不全量替换整行记录                          |
| **向后兼容**       | 保留 `session.metadata["bp_state"]` 写入作为兼容路径，SQLite 恢复优先、session.metadata 回退 |

## 4. 切换与恢复所需数据分析

### 4.1 BPInstanceSnapshot 全部可序列化字段

| 字段                      | 类型               | 用途                                           |
| ----------------------- | ---------------- | -------------------------------------------- |
| `bp_id`                 | str              | BP 模板 ID                                     |
| `instance_id`           | str              | 实例唯一 ID（`bp-{uuid8}`）                        |
| `session_id`            | str              | 所属会话                                         |
| `status`                | BPStatus         | active / suspended / completed / cancelled   |
| `created_at`            | float            | 创建时间戳                                        |
| `completed_at`          | float \| None    | 完成/取消时间戳                                     |
| `suspended_at`          | float \| None    | 挂起时间戳                                        |
| `current_subtask_index` | int              | 当前执行到第几个子任务                                  |
| `run_mode`              | RunMode          | manual / auto                                |
| `subtask_statuses`      | dict\[str, str]  | 各子任务状态，恢复进度 + recovery prompt Steps          |
| `initial_input`         | dict             | 原始输入，恢复时注入 "Original input"                  |
| `subtask_outputs`       | dict\[str, dict] | 各子任务输出，恢复时注入 "Completed outputs"             |
| `context_summary`       | str              | 挂起时 LLM 压缩的上下文摘要（JSON），恢复时构建 recovery prompt |
| `supplemented_inputs`   | dict\[str, dict] | bp\_answer 补充的参数                             |

### 4.2 Session 级辅助状态

| 字段               | 持久化 | 说明                              |
| ---------------- | --- | ------------------------------- |
| `cooldown`       | 是   | 取消后防止重复触发的冷却轮数                  |
| `offered_bps`    | 是   | 已推荐的 bp\_id 集合，防止重复推荐           |
| `pending_switch` | 是   | 待执行的上下文切换（suspend → resume 中间态） |
| `pending_offers` | 否   | 临时性，重启后可丢弃                      |

### 4.3 不持久化

- `bp_config`：运行时引用，恢复时从 config\_loader 回填

## 5. 数据库表设计

### 5.1 bp\_instances 表

```sql
CREATE TABLE IF NOT EXISTS bp_instances (
    instance_id         TEXT PRIMARY KEY,
    bp_id               TEXT NOT NULL,
    session_id          TEXT NOT NULL,
    status              TEXT NOT NULL DEFAULT 'active',
    run_mode            TEXT NOT NULL DEFAULT 'manual',
    current_subtask_index INTEGER DEFAULT 0,
    created_at          REAL NOT NULL,
    completed_at        REAL,
    suspended_at        REAL,
    context_summary     TEXT DEFAULT '',              -- JSON string
    subtask_statuses    TEXT DEFAULT '{}',            -- JSON: {subtask_id: status_str}
    initial_input       TEXT DEFAULT '{}',            -- JSON
    subtask_outputs     TEXT DEFAULT '{}',            -- JSON: {subtask_id: output_dict}
    supplemented_inputs TEXT DEFAULT '{}',            -- JSON: {subtask_id: extra_data}
    updated_at          TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX IF NOT EXISTS idx_bp_inst_session ON bp_instances(session_id);
CREATE INDEX IF NOT EXISTS idx_bp_inst_status  ON bp_instances(status);
CREATE INDEX IF NOT EXISTS idx_bp_inst_bp_id   ON bp_instances(bp_id);
CREATE INDEX IF NOT EXISTS idx_bp_inst_created ON bp_instances(created_at);
```

### 5.2 bp\_session\_state 表

```sql
CREATE TABLE IF NOT EXISTS bp_session_state (
    session_id     TEXT PRIMARY KEY,
    cooldown       INTEGER DEFAULT 0,
    offered_bps    TEXT DEFAULT '[]',       -- JSON array
    pending_switch TEXT,                     -- JSON: {suspended_id, target_id} or NULL
    updated_at     TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### 6.2 字段级更新

| 方法                                                                 | SQL 行为                       | 使用场景                                  |

async def persist_instance(self, instance_id: str) -> None

## 8. 持久化时机汇总

| 事件           | 触发位置                           | 持久化方法                                                           | 更新范围                        |
| ------------ | ------------------------------ | --------------------------------------------------------------- | --------------------------- |
| 创建实例         | `engine.start()`               | `persist_instance`                                              | 全量 INSERT                   |
| subtask 开始   | `engine.advance()`             | `persist_subtask_progress`                                      | index + statuses            |
| subtask 完成   | `engine.advance()`             | `persist_subtask_output` + `persist_subtask_progress`           | output + index + statuses   |
| BP 完成        | `engine.advance()`             | `persist_status_change`                                         | status + completed\_at      |
| 编辑 output    | `handler` / `PUT /edit-output` | `persist_subtask_output` + `persist_subtask_progress`（如有 STALE） | output + statuses           |
| 切换挂起         | `engine.switch()`              | `persist_status_change` x2 + `persist_session_state`            | 两个实例 status + session state |
| 上下文压缩        | `context_bridge`               | `persist_context_summary`                                       | context\_summary            |
| 取消           | `engine.cancel()` / `DELETE /` | `persist_status_change`                                         | status + completed\_at      |
| 补充参数         | `engine.answer()`              | `persist_supplemented_input`                                    | supplemented\_inputs        |
| 切换 run\_mode | `PUT /run-mode`                | `persist_instance`                                              | 全量                          |

## 9. API 端点变更

### 9.1 新增查询端点

| 端点                                                           | 说明                                              |
| ------------------------------------------------------------ | ----------------------------------------------- |
| `GET /api/bp/instance/{instance_id}`                         | 查询单个实例完整信息（含所有 subtask output），内存优先 → SQLite 回退 |
| `GET /api/bp/list?session_id=&status=&bp_id=&limit=&offset=` | 跨 session 查询实例列表，支持多维度筛选和分页                     |

### 9.2 现有端点增强

| 端点                                     | 变更                                                             |
| -------------------------------------- | -------------------------------------------------------------- |
| `GET /api/bp/status`                   | `_ensure_bp_restored` 改为 async，SQLite 优先恢复                     |
| `GET /api/bp/output/{id}/{subtask_id}` | 增加 SQLite 回退查询                                                 |
| `PUT /api/bp/edit-output`              | 增加 `await persist_subtask_output` + `persist_subtask_progress` |
| `PUT /api/bp/run-mode`                 | 增加 `await persist_instance`                                    |
| `DELETE /api/bp/{instance_id}`         | 增加 `await persist_status_change`                               |

### 9.3 恢复流程变更

`_ensure_bp_restored` 改为 async，优先从 SQLite 恢复，回退到 session.metadata：

```
请求到达 → 检查内存有无实例
  ├─ 有 → 直接返回
  └─ 无 → await storage.load_instances_by_session()
       ├─ 有数据 → BPInstanceSnapshot.deserialize() + 回填 bp_config
       └─ 无数据 → 回退 session.metadata["bp_state"]（兼容旧数据）
```

## 13. 验证方案

### 13.1 单元测试 — BPStorage

- 全量 CRUD save → load → deserialize 往返验证
- `update_subtask_output`：只修改目标 subtask，不影响其他 subtask 的 output
- `update_instance_status`：状态字段精准更新，其他字段不变
- `count_instances` 按 session\_id / status 条件统计

### 13.2 单元测试 — BPStateManager

- 各 `persist_*` 方法：内存修改 → SQLite 写入 → 重新 load → 验证数据一致
- `restore_from_db`：不覆盖内存已有实例 + session state 正确恢复

### 13.3 集成测试 — 实时性

- 创建实例 → 立即查 SQLite → 数据存在
- advance 完成 subtask → 立即查 → output 已更新
- edit\_output 修改字段 → 立即查 → 字段值正确、下游状态为 STALE

### 13.4 集成测试 — 查询端点

- `GET /api/bp/list` 跨 session 查询返回多个 session 的实例
- `GET /api/bp/instance/{id}` 返回完整实例详情
- `GET /api/bp/output/{id}/{subtask_id}` 获取 output 用于编辑

### 13.5 集成测试 — 切换恢复

- 创建实例 → 切换挂起 → 清空内存 → restore\_from\_db → 验证 context\_summary、pending\_switch 正确恢复 → 能成功构建 recovery prompt

### 13.6 手动验证

1. `seeagent serve` → 前端发起 BP → 中途查看 `data/agent.db` 中 `bp_instances` 表实时更新
